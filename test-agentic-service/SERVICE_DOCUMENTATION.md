# 저기어때 - 여행 계획 에이전틱 서비스 상세 문서

## **1. 프로젝트 개요 (2026-03-24 업데이트)**

### **1.1 프로젝트 기획 배경**
- 여행 계획 초기에 사용자가 너무 많은 선택지를 동시에 마주해 시작 장벽이 높다는 문제를 해결한다.
- 기존 방식의 한계:
  - 여행지 추천, 항공권 검색, 일정 수립을 분절된 서비스에서 개별 수행
  - 사전 조사(블로그, 후기, 가격 비교)에 시간 과다 소요
  - 정보는 많지만 의사결정을 직접 도와주는 구조화 결과는 부족
- Agent 기반 접근으로 입력 해석, 단계별 생성, 결과 선택 중심 UX를 제공한다.

### **1.2 핵심 아이디어 / Value Proposition**
- 여행 테마 기반 여행지 자동 추천
- 추천 여행지 왕복 항공권 자동 탐색
- 여행 일정/예산 초안 자동 생성
- 단순 검색이 아니라 의사결정에 바로 사용할 수 있는 구조화 결과를 제공한다.

### **1.3 대상 사용자 / 기대 UX**
- 여행 초안 작성을 부담스러워하는 일반 사용자
- 빠르게 실행 가능한 여행 시작점을 원하는 사용자
- 입력(주제/조건) -> 자동 생성(도시/항공/일정/예산) -> 검토/수정의 흐름을 제공한다.

## **2. 기술 구성 업데이트 (기준일: 2026-03-24 커밋 반영)**

### **2.1 Prompt Engineering 전략**
- Agent별 역할 분리형 프롬프트를 유지하고 출력 책임을 JSON 스키마로 고정했다.
- 각 Agent 프롬프트에 Few-shot 예시를 포함해 형식 일관성을 강화했다.
- 공통 응답 가드(`response_guard`)를 통해 코드블록 제거/JSON 파싱을 표준화했다.
- 필수 키 누락 시 JSON 리페어를 1회 수행해 실패율을 낮췄다.
- `rationale` 필드를 공통 도입해 추천/선정/예산 근거를 짧게 명시하도록 했다.

### **2.2 LangChain / LangGraph 기반 Agent 구조**
- Supervisor + Worker(도시추천/항공권/일정) Multi-Agent를 LangGraph 상태 그래프로 운영한다.
- Supervisor는 항공권 미가용 시 다음 도시 재검색 분기와 재시도 제어를 담당한다.
- `TravelState`의 `decision_memory`, `constraints_memory`를 누적해 후속 Agent 프롬프트에 재주입한다.

### **2.3 RAG 구성**
- 목적: 최신 정보 반영 + 의사결정 연속성 강화.
- 하이브리드 RAG:
  1. 벡터 검색(FAISS) 우선
  2. 결과 부족 시 DuckDuckGo 웹 검색 fallback
- `search_web`를 LangChain Tool(`@tool`)로 변환하고 Agent가 `bind_tools`로 호출한다.
- 검색은 사전 고정 호출이 아니라 Agent 실행 흐름(Action) 내부 Tool 호출로 통합됐다.

### **2.4 서비스 개발 및 패키징**
- UI: Streamlit
- Backend: FastAPI (`GET /api/health`, `POST /api/plan`)
- 환경설정: `.env` 기반 키 관리
- 실행 중심은 로컬(venv)이며, 현재 버전에서는 컨테이너화 산출물을 제외했다.

### **2.5 선택적 확장 기능**
- 선반영:
  1. Tool Calling(`bind_tools`)
  2. ReAct 실행 정책 표준화(공통 `tool_runner`)
  3. 상태 메모리 기반 연속 의사결정
  4. FastAPI API 인터페이스
- 향후:
  1. 외부 항공/숙소 API(Function Calling 확장)
  2. 장기 메모리 저장소(DB) 연동
  3. 멀티소스 RAG 재랭킹 고도화
  4. A2A 구조로 Supervisor 의존도 완화

## **2. 4.2 필수 기술 요소 구현 현황**

| 항목 | 구현 상태 | 구현 내용 | 미구현/보강 필요 |
|------|----------|-----------|------------------|
| Prompt Engineering | 상당 구현 | 역할 기반 system prompt, 입력 구조화, JSON 출력 강제, Few-shot 예시, 필수 키 누락 시 JSON 보정 1회 | CoT/추론 단계의 정교한 표준화 |
| LangChain/LangGraph Agent | 구현됨 | Multi-Agent, LangGraph Supervisor 분기, Tool Calling(`bind_tools`), 상태 메모리 누적/재사용, ReAct 실행 정책 표준화 | - |
| RAG | 상당 구현 | 데이터 전처리/청킹, 임베딩, FAISS, 하이브리드 검색(벡터+웹) | 고도화 항목(재랭킹/멀티소스 등)만 잔존 |
| 서비스 개발/패키징 | 구현됨 | Streamlit UI, FastAPI 백엔드(`GET /api/health`, `POST /api/plan`) | - |

### **2.1 근거 모듈**
- Prompt/Agent: `app/workflow/agents/*`
- LangGraph 분기: `app/workflow/graph.py`, `app/workflow/agents/supervisor_agent.py`
- Tool Calling: `app/workflow/agents/tool_runner.py`
- RAG(FAISS/하이브리드): `app/retrieval/vector_store.py`, `app/retrieval/search_service.py`, `app/retrieval/knowledge_loader.py`

### **2.2 미구현 체크리스트 (우선순위)**
- [x] P1 Prompt: Few-shot + 일관성 보정 템플릿
- [x] P1 Memory: 상태 메모리 누적/재사용 고도화
- [x] P1 FastAPI 백엔드: `GET /api/health`, `POST /api/plan`
- [x] P2 ReAct: Tool 실행 흐름 포맷 표준화

### **2.3 Prompt 보강 내용**
- Agent A/B/C system prompt에 Few-shot 예시 추가
- `rationale` 필드를 공통 출력 스키마에 포함
- JSON 파싱 시 필수 키 누락을 감지하면 LLM 기반 JSON 리페어를 1회 수행

### **2.4 Memory 보강 내용**
- `TravelState`에 `decision_memory`, `constraints_memory`를 추가해 상태 메모리 누적
- Supervisor 분기 시 미가용 사유/도시 전환 내역을 메모리에 기록
- Agent B/C 프롬프트에 최근 의사결정/제약 메모리를 주입해 재시도 품질 보강

### **2.5 FastAPI 백엔드 보강 내용**
- 신규 엔트리: `app/api/main.py`
- 헬스체크: `GET /api/health`
- 플래너 실행: `POST /api/plan`
- 요청/응답 스키마를 Pydantic으로 명시해 UI와 분리된 백엔드 실행 가능

### **2.6 ReAct 표준화 내용**
- `tool_runner`에 ReAct 정책(system rule)을 추가해 tool call 루프를 명시적으로 통일
- 도구 실행 로그를 `action/observation` 구조로 기록해 디버깅 일관성 확보
- 최종 응답은 기존 JSON 스키마를 유지하도록 제약

**사용자**: 직장인 김미래 (30대, 5일 휴가)

**상황**:

- 서울 출발, 해외 여행지 미정

**서비스 이용 과정**:

1. **입력 단계**
   ```
   👤 사용자 입력:
   - 여행 주제: "해변에서 휴양"
   - 여행 일수: 5일
   - 예산: 2,000,000원
   - 출발 도시: 서울
   - RAG 활성화: ✅
   ```

2. **Agent A: 도시 추천**
   ```
   🌍 도시 추천 에이전트 실행
   
   [웹 검색]
   - 쿼리: "해변에서 휴양 여행 추천 도시 해외"
   - DuckDuckGo에서 실시간 여행 정보 수집
   - 발리, 푸켓, 세부 등 인기 해변 도시 정보 확보
   
   [AI 분석 및 추천]
   - 검색 결과 + LLM 지식 통합
   - 추천 결과:
     1. 발리, 인도네시아 - "완벽한 해변과 힐링 스파가 유명"
     2. 푸켓, 태국 - "저렴한 가격에 아름다운 해변 즐기기"
     3. 세부, 필리핀 - "에메랄드빛 바다와 스노클링 명소"
   
   [자동 선택]
   - 첫 번째 도시 '발리' 자동 선택
   ```

3. **Agent B: 항공권 검색**
   ```
   ✈️ 항공권 검색 에이전트 실행

   - 쿼리: "서울 발리 항공권 운항 여부 예약 가능 여부 평균 가격 항공사"
   - 실제 항공권 가격 정보 수집
   - 주요 항공사 및 시즌별 가격 정보 확보
   
   [AI 분석 및 생성]
   - 출발일: 2026-02-26 (오늘 + 30일)
   - 귀국일: 2026-03-03 (출발일 + 5일)
   - 항공권 정보:
     * 항공사: 대한항공
     * 출발 공항: ICN (인천)
     * 도착 공항: DPS (발리)
     * 가격: 850,000원 (왕복)
   ```

4. **Agent C: 일정 + 예산 계획**
   ```
   📅 일정 계획 에이전트 실행
   
   [웹 검색]
   - 쿼리: "발리 해변휴양 여행 일정 추천 명소 맛집"
   - 현지 명소, 맛집, 액티비티 정보 수집
   - 우붓, 스미냑, 울루와투 등 핫플레이스 확보
   
   [AI 분석 및 계획]
   - Day 1: 공항 도착 → 스미냑 해변 산책 → 호텔 체크인
   - Day 2: 우붓 원숭이 숲 → 테갈랄랑 라이스테라스 → 전통 발리 마사지
   - Day 3: 스노클링 투어 → 해변 휴식 → 선셋 감상
   - Day 4: 울루와투 사원 → 케착 댄스 관람 → 징글 해변
   - Day 5: 쇼핑 → 공항 출발
   
   - 예산 분배:
     * 항공권: 850,000원
     * 숙소: 600,000원 (1박 120,000원 × 5박)
     * 식비: 350,000원 (1일 70,000원 × 5일)
     * 기타: 200,000원 (투어, 쇼핑, 교통)
     * 총 예산: 2,000,000원
   ```

5. **결과 확인**

   [탭 1: 추천 도시]

   - 푸켓, 태국
   - 세부, 필리핀
   
   [탭 2: 항공권]
   - 대한항공, ICN → DPS
   - 2026-02-26 출발, 2026-03-03 귀국
   - 850,000원
   
   [탭 3: 일정]
   - Day 1 ~ Day 5 상세 일정
   
   [탭 4: 예산]
   - 항목별 예산 분배
   - 총 2,000,000원
   
   [사이드바: 실행 로그]

   - LLM 호출 및 응답 로깅

### **3.2.1 Supervisor 분기 로직 (신규)**

항공권 단계는 이제 고정 순차 흐름이 아니라, `Supervisor`가 `TravelState`를 보고 조건 분기를 수행합니다.

```
[분기 입력 상태]
- flight_available (bool)
- flight_unavailability_reason (str | None)
- selected_city_index (int)
- flight_search_attempts (int)
- max_flight_search_attempts (int)

[분기 규칙]
1) 항공권 가용(true):
   FLIGHT_SEARCH -> ITINERARY_PLANNER

2) 항공권 미가용(false) + 재시도 가능:
   - 추천 도시에 다음 후보가 있으면 selected_city를 다음 도시로 전환
   - FLIGHT_SEARCH 재실행

3) 항공권 미가용(false) + 재시도 한도 초과:
   FLIGHT_SEARCH -> ITINERARY_PLANNER (보수적 fallback)
```

가용성 판정 기준:
- 필수 항공권 필드 누락 여부 (`departure/arrival/date/airline/price`)
- 가격 유효성 (`price > 0`)
- 검색 컨텍스트 내 미가용 시그널 (`예약 불가`, `운항 없음`, `no flights` 등)

```
시간: 0-15초

[단계 1] 입력 로깅 (0.1초)
├─ 로거: CityRecommenderAgent
├─ 내용: 여행 주제, 일수, 예산 출력

├─ 쿼리 생성: "해변에서 휴양 여행 추천 도시 해외"
├─ DuckDuckGo API 호출
│   ├─ 시도 1: 성공 또는 Rate Limit
│   ├─ 시도 2: 2초 대기 후 재시도
│   └─ 시도 3: 4초 대기 후 재시도
├─ 결과: 3개 웹페이지 (제목, 본문, URL)
└─ 컨텍스트 포맷팅: 약 1,500자

[단계 3] LLM 호출 (5-10초)
├─ 프롬프트 구성:
│   ├─ System: "당신은 전문 여행 컨설턴트입니다..."
│   └─ Human: "검색 결과: ... \n여행 주제: ..."
├─ OpenAI GPT-4 API 호출
├─ 응답: JSON 형식 (도시 2~3개)
└─ 토큰: 약 500 tokens

[단계 4] JSON 파싱 (0.1초)
├─ 코드 블록 제거 (```json```)
├─ JSON.loads()
├─ 결과 추출: recommended_cities
└─ 첫 번째 도시 자동 선택

[단계 5] 출력 로깅 (0.1초)
├─ 추출된 도시 목록 출력
├─ 선택된 도시 표시

```
시간: 15-30초

[단계 1] 입력 로깅 + 날짜 계산 (0.1초)
├─ 선택된 도시 확인
├─ 출발일 = today + 30일
└─ 귀국일 = 출발일 + travel_days

├─ 쿼리: "서울 발리 항공권 운항 여부 예약 가능 여부 평균 가격 항공사"

├─ 검색 결과 + 날짜 정보
├─ GPT-4가 항공권 정보 생성

├─ 필수 필드/가격 유효성 검사

```

```
시간: 30-50초

[단계 1] 입력 로깅 (0.1초)
├─ 도시 + 항공권 정보 확인
└─ 여행 일수 확인

[단계 2] RAG 검색 (3-8초)
├─ 쿼리: "발리 해변휴양 여행 일정 추천 명소 맛집"
├─ 검색 결과 4개 (더 많은 정보 필요)
└─ 명소, 맛집, 액티비티 정보

[단계 3] LLM 호출 (10-15초)
├─ 가장 긴 프롬프트 (검색 결과 + 도시 + 항공권)
├─ GPT-4가 일정 + 예산 생성
└─ JSON: itinerary + budget_breakdown

[단계 4] 파싱 + 로깅 (0.1초)
├─ 일정 배열 추출
├─ 예산 객체 추출
└─ State.completed = true
```

#### **Phase 5: UI 렌더링**
```
시간: 50-52초

[Streamlit 렌더링]
├─ st.success("여행 계획이 완성되었습니다!")
├─ st.rerun() 호출
├─ 탭 UI 생성
│   ├─ 탭1: 추천 도시 (Expander)
│   ├─ 탭2: 항공권 (Metrics)
│   ├─ 탭3: 일정 (Expander)
│   └─ 탭4: 예산 (Metrics)
└─ 사이드바: 로그 (st.text)

[총 소요 시간]
약 50초 (RAG ON)
약 20초 (RAG OFF)
```

### **확장 가능성**

```
현재 구조 (MVP)
    │
    ├─ [Phase 2] 병렬 처리
    │   └─ Agent A → (Agent B || Agent C) 동시 실행
    │
    ├─ [Phase 3] 사용자 선택
    │   └─ 도시 선택 UI → Agent B, C 재실행
    │
    ├─ [Phase 4] 추가 에이전트
    │   ├─ Agent D: 숙소 추천 (Booking.com API)
    │   ├─ Agent E: 맛집 추천 (Google Places API)
    │   └─ Agent F: 날씨 정보 (OpenWeather API)
    │
    ├─ [Phase 5] 데이터베이스
    │   └─ 여행 계획 저장/불러오기 (SQLite → PostgreSQL)
    │
    └─ [Phase 6] 고급 RAG
        ├─ Vector DB (FAISS, Pinecone)
        ├─ 문서 임베딩 및 유사도 검색
        └─ 멀티소스 검색 (DuckDuckGo + Bing + Google)
